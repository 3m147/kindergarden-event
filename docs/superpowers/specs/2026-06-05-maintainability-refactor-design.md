# 유지보수성 리팩터링 설계

## 목표

현재 프로젝트의 기능, API 계약, 데이터 저장 형식, CSS 클래스, UI 배치와 사용자 경험을 변경하지 않고 내부 코드 구조만 정리한다.

이번 작업은 새로운 기능 개발이나 디자인 수정이 아니다. 파일 책임을 명확히 나누고 중복 정의를 제거해 이후 수정 시 영향 범위를 줄이는 것이 목적이다.

## 절대 보존 항목

- 모든 화면의 JSX 렌더링 순서
- 모든 Tailwind CSS 클래스 문자열과 조건
- 교사·관리자·학생 체크 화면의 사용자 흐름
- API 경로, HTTP 메서드, 요청 및 응답 데이터 형식
- 로컬스토리지 키와 저장 데이터 형식
- 공과 활성화 날짜 계산과 유치부 체크 해제 조건
- 로그인, 공지, 사진, 계획표, 머릿돌, 공과 영상, 정보 수정 기능
- 기존 커밋되지 않은 사용자 작업

## 현재 구조 분석

### 프론트엔드

유지보수 위험이 세 대형 컴포넌트에 집중되어 있다.

- `StartView.tsx`: 로그인, 홈, 공지, 자료, 내 정보 화면을 함께 담당
- `AdminDashboardView.tsx`: 대시보드와 모든 관리자 자료 관리 기능을 함께 담당
- `TeacherCheckView.tsx`: 학생 목록, 체크 상태, 여러 모달과 네비게이션을 함께 담당

또한 공과 제목, 유치부 활동 종류, 총 과 수와 같은 도메인 상수가 관리자와 교사 화면에 중복 정의되어 있다.

### 백엔드

전체 구조는 비교적 단순하며 대규모 분리가 필요하지 않다. `RecitationService` 내부에서 반복되는 학생 상태 조회와 DTO 변환 로직만 작은 메서드로 정리할 가치가 있다.

## 선택한 접근 방식

### 보존 중심 구조 분리

하위 컴포넌트와 상수를 기능별 파일로 이동하되, 컴포넌트의 마크업과 CSS 클래스는 그대로 유지한다. 상위 컴포넌트는 상태와 이벤트 연결을 담당하며 기존 동작 흐름을 유지한다.

복잡한 신규 훅, 상태 관리 라이브러리, 디자인 시스템 도입은 하지 않는다. 현재 구조에 새로운 추상화를 과도하게 추가하면 기능 회귀 위험이 커지기 때문이다.

## 목표 구조

```text
frontend/
  components/
    start/
      home-components.tsx
      notice-components.tsx
      info-components.tsx
      resource-components.tsx
    admin/
      resource-managers.tsx
      dashboard-components.tsx
    check/
      check-components.tsx
      check-modals.tsx
  constants/
    kindergarten.ts
```

실제 분리 과정에서 파일 크기와 의존성을 확인하여 파일을 더 합치거나 나눌 수 있다. 다만 역할 경계는 위 구조를 따른다.

## 단계별 변경 범위

### 1. 공용 도메인 상수 통합

다음 중복 정의를 공용 상수 파일로 이동한다.

- 전체 유치부 과 수
- 유치부 활동 종류
- 분기별 공과 범위
- 공과 제목

상수의 값, 배열 순서, 키 문자열은 변경하지 않는다.

### 2. StartView 하위 컴포넌트 분리

다음 순수 렌더링 컴포넌트를 별도 파일로 이동한다.

- 홈 요약과 사진 슬라이더
- 공지함, 공지 설정, 공지 상세
- 내 정보 조회와 수정 관련 컴포넌트
- 계획표, 공과 영상 목록, 하단 네비게이션

상태와 이벤트 핸들러는 우선 `StartView`에 유지하여 동작 변경 위험을 줄인다.

### 3. AdminDashboardView 하위 컴포넌트 분리

다음 관리자 렌더링 컴포넌트를 별도 파일로 이동한다.

- 공지 및 주간 사진 관리
- 머릿돌 관리
- 계획표 관리
- 공과 영상 관리
- 통계 카드와 그래프

파일 업로드, 로컬스토리지 저장, 토스트 상태는 상위 컴포넌트에 유지한다.

### 4. TeacherCheckView 하위 컴포넌트 분리

다음 렌더링 컴포넌트를 별도 파일로 이동한다.

- 학생 카드와 아바타
- 하단 네비게이션과 확인 다이얼로그
- 암송, 퀴즈, 유치부 체크 모달
- 범례와 진행률 표시

학생 상태와 API 토글 로직은 상위 컴포넌트에 유지한다.

### 5. 백엔드 반복 로직 정리

`RecitationService` 안에서 반복되는 다음 로직을 private 메서드로 정리한다.

- 특정 날짜의 학생 상태 조회
- 학생 DTO 검색
- 레코드 상태 맵 변환

컨트롤러 경로, 서비스 공개 메서드 시그니처, DTO 구조는 유지한다.

## 오류 처리

기존 오류 처리 방식과 사용자 메시지를 그대로 유지한다. 리팩터링 과정에서 새로운 오류 문구, 토스트, 폴백 동작을 추가하지 않는다.

## 검증 전략

각 단계 완료 후 다음 검증을 수행한다.

1. `git diff --check`
2. 프론트엔드 `npm run build`
3. 백엔드 `mvn test`
4. 프론트와 백엔드 HTTP 응답 확인
5. 가능할 경우 주요 화면의 조회와 전환을 브라우저에서 확인

각 단계에서 JSX와 Tailwind 클래스가 이동 외에 변경되지 않았는지 diff를 확인한다.

## 제외 범위

- 새로운 기능 추가
- UI 또는 CSS 개선
- API 및 DB 스키마 변경
- 인증 방식 변경
- 로컬스토리지 기반 자료 관리 방식 교체
- 테스트 프레임워크 신규 도입
- 기존 미커밋 변경의 삭제 또는 되돌리기

## 완료 기준

- 기존 빌드와 백엔드 테스트가 모두 통과한다.
- 화면 기능과 UI가 변경되지 않는다.
- 중복 도메인 상수가 하나의 위치에서 관리된다.
- 세 대형 프론트엔드 컴포넌트의 렌더링 책임이 기능별 파일로 분리된다.
- 백엔드 반복 로직이 명확한 private 메서드로 정리된다.
